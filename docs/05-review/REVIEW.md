# 工程5: セルフレビュー

## 概要

pixelmatch検証（工程6）の前に、実装コードを計画書・規約と照合する。
不備があれば視覚検証に進まず即修正し、再レビューする。
この工程はAIが自動進行する（ユーザーの承認不要）。

---

## 目的

- コーディング規約違反を視覚検証前に検出・修正する
- 計画書との乖離を検出し、意図しない実装を防ぐ
- Figma値の正確性を保証する
- 無駄なpixelmatch反復を減らす

---

## チェックリスト

### A. 計画書との照合

計画書: `docs/mcp-log/{page}/{section}-plan.md`

- [ ] **HTML構造の一致**: 計画書のネスト図と実装HTMLの構造が一致しているか
- [ ] **クラス名の一致**: 計画書で決定したクラス名がそのまま使われているか
- [ ] **SCSS方針の一致**: 計画書のレイアウト手法（flex/grid/通常フロー）が守られているか
- [ ] **レスポンシブ方針の一致**: ブレークポイントごとの変更が計画通りか
- [ ] **画像の一致**: 計画書の画像リストの全画像が取得・配置されているか

### B. パターンライブラリとの照合

参照: [パターンライブラリ](../patterns/README.md)

- [ ] **HTML構造がパターンテンプレートと整合**: 大幅な構造逸脱がないか
- [ ] **SCSSがパターンテンプレートと整合**: レイアウト手法の不整合がないか
- [ ] **パターンの「よくある失敗」に該当していないか**: 各パターンファイルの失敗表を確認

### C. CODING_RULES.md 全項目の照合

参照: [CODING_RULES.md](../CODING_RULES.md)

#### SCSS規約
- [ ] `@use "foundation" as *;` のみがインポートされているか（複数行のインポートは不要）
- [ ] `style.scss` に手動追記していないか
- [ ] FLOCSS + BEM記法のクラス名になっているか（`.p-{section}__{element}`）
- [ ] すべての数値が `rem()` 関数で囲まれているか
- [ ] `letter-spacing` は `em` 単位で記述されているか（rem()禁止、px禁止）
- [ ] letter-spacing の計算が正しいか（`letter-spacing(px) ÷ font-size(px) = em値`）
- [ ] ネスト（`&__element`）が使用されていないか（フラット記述）
- [ ] メディアクエリが各クラス内に記述されているか（ファイル末尾にまとめていないか）
- [ ] カラーコードが直接書かれていないか（変数を使用しているか）
- [ ] フォントファミリーが直接書かれていないか（変数を使用しているか）
- [ ] `position: absolute` が原則禁止ルールに違反していないか
- [ ] `padding-bottom` / `padding-left` を使用していないか（上方向・右方向に統一）
- [ ] `margin: 0 auto` ではなく `margin-inline: auto` を使用しているか
- [ ] PCコーディングで縦余白に `flex-direction: column` + `gap` を使っていないか（margin-top推奨）
- [ ] 縦余白でタイトル以外の要素に `margin-bottom` を使っていないか（タイトルのみ `margin-bottom`、それ以外は `margin-top`）
- [ ] 横並びflexのアイテム間余白に `margin-left` / `margin-right` を使っていないか（`gap` を使用すること。`margin-left: auto` / `margin-right: auto` の配置用途は除く）
- [ ] 固定幅に `max-width: 100%` がセットされているか
- [ ] `border-radius: 100%` を使用しているか（`9999px` は不可）
- [ ] `object-fit: contain` を使用していないか

#### HTML規約
- [ ] セマンティックHTMLタグ（header, nav, main, section, article, footer）が適切に使用されているか
- [ ] 見出し構造が階層順（h1 → h2 → h3）になっているか
- [ ] h1は1ページに1つのみか
- [ ] ナビゲーション・リストには ul/ol/li タグが使用されているか
- [ ] すべての `<img>` に `alt` 属性があるか
- [ ] リンク要素が `<a>` タグになっているか
- [ ] 単体pタグをdivで囲っていないか

#### 画像規約
- [ ] SVGアイコンが自作されていないか（Figmaからダウンロードしているか）
- [ ] 画像パスが実在するか
- [ ] 複合背景が適切に処理されているか（3層以上→1枚画像書き出し）

#### JS依存・動作チェック
- [ ] HTMLで読み込んでいるJSライブラリの依存関係は正しいか（例: script.jsがjQueryを使うなら、jQueryのscriptタグがそれより前にあるか）
- [ ] ブラウザコンソールにエラーが出ていないか（Playwrightの `console_logs(type: "error")` で確認）
- [ ] Swiperなどのインタラクティブ要素が実際に動作するか（ボタンクリックでスライドが切り替わるか）

### D. Figma値の正確性

MCPログ: `docs/mcp-log/{page}/design-context.md`

- [ ] font-size がFigma値と一致しているか（小数点含む）
- [ ] line-height がFigma値と一致しているか
- [ ] letter-spacing がFigma値から正確に計算されているか
- [ ] margin / padding がFigma値と一致しているか
- [ ] color がFigma値と一致しているか（変数経由）
- [ ] width / height がFigma値と一致しているか
- [ ] border-radius がFigma値と一致しているか
- [ ] gap がFigma値と一致しているか

#### レイアウト計算の検証（必須）

`justify-content: space-between` や `gap` で余白を作るレイアウトでは、**実効的なコンテナ幅と子要素の合計幅の差分を計算**して、意図した余白が生まれるか検証する。

```
検証手順:
1. Figmaメタデータからコンテナ幅と子要素の幅を取得
2. 実装のCSSで同じ計算をして、コンテナのcontent areaを算出
3. 祖先要素のpadding-inline / max-widthが実効幅に与える影響を確認
4. 計算結果がFigmaと一致するか確認

例:
Figma: コンテナ1095.73px、子要素690+360=1050px → gap=45.73px ✅
実装: inner(1240px) - padding(25*2) = 1190px → item(1190px) - padding(72*2) = 1046px → gap=0 ❌
      ↑ 祖先のpadding-inlineが実効幅を圧縮していた
```

**よくある罠**: 親コンテナの`padding-inline`が子の実効幅を縮め、`space-between`のgapが消える。Figmaの数値だけでなく、CSS上の計算チェーンを通して実効幅を検証すること。

### E. クイック視覚チェック（必須）

コードレビュー（A〜D）完了後、**実際の画面とFigmaデザインを構造的に比較する**。
pixelmatchのような精密比較ではなく、「一目見てわかる」レベルの差異を検出する目的。

**重要: 「前より良くなった」ではなく「Figmaと一致しているか」で判断すること。**
修正前→修正後の比較では確認バイアスが生じる。必ずFigmaデザインとの直接比較を行う。

#### 手順

1. **ビルド実行**: `npx gulp build` でSCSSをコンパイル
2. **実装スクリーンショット撮影**: Playwrightで対象セクションのスクリーンショットを撮影
3. **Figmaスクリーンショット取得**: Figma MCPの `get_screenshot` で同じセクションのデザイン画像を取得
4. **要素インベントリの作成と突き合わせ**（以下の手順に厳密に従う）:

#### STEP 4a: Figma側の要素インベントリ

Figmaスクリーンショットに見える要素を**すべてリストアップ**する。

```
例:
Figma要素リスト:
- 背景: 薄いグレー(#f6f6f6)、ほぼ無地（ごく微かなパターンのみ）
- テキスト: "Message"ラベル、見出し、本文4段落
- 画像1: 人物写真（五角形マスク）
- 画像2: 地球（ダイヤモンドマスク）
- その他の視覚要素: なし
```

#### STEP 4b: 実装側の要素インベントリ

実装スクリーンショットに見える要素を**すべてリストアップ**する。

```
例:
実装要素リスト:
- 背景: 薄いグレー + ネットワークパターンが視認できる ← Figmaと異なる！
- テキスト: "Message"ラベル、見出し、本文4段落
- 画像1: 人物写真（五角形マスク）
- 画像2: 地球（ダイヤモンドマスク）
- その他の視覚要素: 背景パターンに地球のようなモチーフが含まれている ← 差異！
```

#### STEP 4c: 差異の特定

2つのリストを突き合わせ、差異を**明示的に列挙**する。

```
差異レポート:
1. [NG] 背景: Figmaはほぼ無地 → 実装はネットワークパターンが目立つ
2. [OK] テキスト: 一致
3. [OK] 画像の数: 一致（2枚）
4. [NG] 背景パターンに地球のようなモチーフが見える → 実質的に地球画像が2つ表示されている
```

**差異が0件でなければNGとする。**

#### チェック項目（STEP 4cの差異レポートに基づいて判定）

- [ ] **要素の数**: 画像・テキストブロック・ボタン等の数がFigmaと一致しているか（重複や欠落がないか）
- [ ] **レイアウトの大枠**: 左右・上下の配置関係がFigmaと一致しているか
- [ ] **背景の見た目**: 背景色・背景パターンがFigmaと大きく異なっていないか
- [ ] **画像のサイズ感**: 各画像の大きさの比率がFigmaと大きく異なっていないか
- [ ] **テキストの可読性**: テキストが背景に埋もれたり、要素と重なったりしていないか

#### なぜインベントリ方式が必要か

「見て確認する」だけでは**確認バイアス**に陥る。
特に修正直後は「良くなったはず」という思い込みが働き、残存する差異を見落とす。
要素を1つずつリストアップして突き合わせることで、バイアスを排除し差異を確実に検出する。

工程4（コーディング）中の目視確認は「コードを書きながら」のため見落としやすい。
工程6（pixelmatch）は重い処理でオンデマンド実行。
この工程で「一目でわかる明らかな差異」を検出し、工程6の前に修正することで、
無駄なトークン消費と手戻りを防ぐ。

#### NGの場合

差異が見つかった場合:
1. 差異レポートに基づいて原因を特定（画像の重複、ブレンドモード未適用、要素の欠落等）
2. 即修正
3. 再ビルド → 再スクリーンショット → **STEP 4a〜4cを再実行**
4. 差異が0件になるまで繰り返す

---

## 判定

### 合格: A〜Eの全チェック項目がOK

→ コーディング完了。ユーザーに報告する。
→ 工程6（視覚的検証）はユーザーがオンデマンドで指示する。

### 不合格: 1つ以上のNG

→ 即修正 → 再レビュー（NGだったチェックリストを再実行）

**明らかな差異がある状態でコーディング完了としない。**

---

## 注意事項

- セルフレビューは1セクション完了ごとに実施する
- 複数セクション一括指示の場合も、各セクション完了時にレビューする
- レビュー結果は `docs/mcp-log/{page}/{section}-review.md` に保存してもよい（任意）
