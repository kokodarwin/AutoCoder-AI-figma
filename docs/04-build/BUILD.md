# 工程4: コーディング

## 概要

Figma MCP経由でデザインを取得し、**セクションごとに** HTMLとSCSSを順番にコーディングする。
各セクションで必要な部分だけを取得し、コンテキストを適切なサイズに保つ。

**コーディング規約:** [CODING_RULES.md](../CODING_RULES.md) を必ず参照すること。

---

## コーディング前の参照（必須）

コードを書き始める前に、以下を必ず確認すること:

### 1. 実装計画の確認

`docs/mcp-log/{page}/{section}-plan.md` に保存された計画書を開き、以下を確認:
- HTML構造のネスト図
- SCSS方針（レイアウト手法、レスポンシブ戦略）
- 画像リスト
- リスク箇所の対処方針

### 2. パターンライブラリの参照

計画書で選定されたパターンのテンプレートを参照する:
- [パターンライブラリ一覧](../patterns/README.md)
- HTML構造テンプレートをベースにカスタマイズ
- SCSSテンプレートのレイアウト手法を踏襲
- 「よくある失敗」を事前にチェック

### 3. 過去実装例の参照

類似セクションの実装例がある場合は参照する:
- [実装例一覧](../examples/README.md)
- クラス名のパターンを参考にする
- 意思決定の記録から判断理由を理解する

## 重要ルール

- **position: absolute; の使用は原則禁止**
- **要素の並び順や位置は、Figmaのノード情報ではなく、スクリーンショットの見た目を参照して組み立てる**
- Flexbox や Grid などの通常のレイアウト手法を使用する
- **MCPのコード情報はあくまで構造の参考。スタイリングの詳細は必ずスクリーンショットで確認すること**

### MCPログ参照ファースト（修正・調整時）

**修正・調整セッションでは、Figma MCPに新規アクセスする前に `docs/mcp-log/` を必ず確認する。**

```
IF 修正・調整タスク:
  1. docs/mcp-log/{page}/ 内のログファイルを確認
  2. 必要な情報がログに存在する場合 → ログの値を使用（MCP呼び出し不要）
  3. ログに存在しない情報のみ → Figma MCPで取得
```

**理由**: 初回コーディング時に取得済みのデザイン情報を再取得すると、無駄なトークンを大量消費する。MCPログにはHEXコード、px値、レイアウト情報が正確に記録されているため、修正時はこれを参照するだけで十分。

### 複雑なSVGのダウンロード禁止

複雑なSVGイラスト（マスク付き、複数VECTORレイヤー構成）は、Figma MCPでダウンロード・合成を試みない。
ユーザーに手動エクスポートを依頼する。詳細は [ASSETS.md](../01-assets/ASSETS.md#複雑なsvgの取り扱いトークン節約) を参照。

### 小さな装飾要素の優先度

修正セッションでは、**レイアウト・構造の修正を優先**し、小さな装飾要素（数十px程度のイラスト・アイコン等）は後回しにする。

```
優先度:
1. レイアウト崩れ（位置ズレ、幅の不一致、段落ち等）
2. 色・フォントの不一致
3. 画像の差し替え・サイズ調整
4. 小さな装飾要素（イラスト、アイコン等）← 最後に対応
```

**理由**: 小さな装飾SVGの取得・合成に大量のトークンを消費すると、重要な修正に使えるトークンが不足する。

## 指示パターン: 複数セクション一括指示

複数セクションのnode-idを一度に渡すことで、全セクションを自動で順次処理できる。
1セクションずつ指示を出す必要はない。

### 指示の出し方

```
以下のセクションを順番にコーディングしてください。

1. ヘッダー: node-id=xxx-xxx
2. FV: node-id=xxx-xxx
3. About: node-id=xxx-xxx
4. Service: node-id=xxx-xxx
5. Contact: node-id=xxx-xxx
6. Footer: node-id=xxx-xxx

fileKey: xxxxxxxx
```

### AIの処理フロー

```
TaskCreateで全セクションのタスクを登録
  ↓
セクション1:
  計画書確認 → サブエージェントでMCP取得・mcp-log保存 → mcp-logをRead → HTML/SCSS作成 → スクショで目視確認 → 明らかな問題のみ修正
  ↓ 完了
セクション2:
  計画書確認 → サブエージェントでMCP取得・mcp-log保存 → mcp-logをRead → HTML/SCSS作成 → スクショで目視確認 → 明らかな問題のみ修正
  ↓ 完了
... 全セクション完了まで自動で繰り返す ...
  ↓
完了通知
（※ セルフレビューは工程5、ピクセルパーフェクト検証は工程6でそれぞれ実施）
```

### コンテキストウィンドウの目安

| ページの複雑さ | 1会話で処理できる目安 | 対応 |
|---------------|---------------------|------|
| シンプル（テキスト+画像程度） | 6〜8セクション | 一括でOK |
| 中程度（カード、グリッドあり） | 4〜5セクション | 一括でOK |
| 複雑（アニメーション、複雑なレイアウト） | 2〜3セクション | 分割推奨 |

コンテキストが圧迫された場合は、途中で「ここまで完了」と報告する。
mcp-logとコードは保存済みなので、次の会話で残りから再開できる。

## 手順

### 1. Figmaからデザイン構造を確認（サブエージェント経由）

**⚠️ コンテキスト隔離ルール（[PIPELINE.md](../PIPELINE.md#コンテキスト隔離ルール必須) 参照）に従い、Figma MCPの呼び出しは必ず Task ツール（サブエージェント）経由で実行する。**

```
メインエージェントの手順:
  1. Task ツールでサブエージェントを起動し、Figmaデータ取得を委託
  2. サブエージェントが get_design_context を実行し、結果を docs/mcp-log/ に保存
  3. メインエージェントは保存されたファイルを Read で読んで作業する
```

**サブエージェントへの指示に含めること:**
- 対象の fileKey と nodeId
- `get_design_context` でデザイン構造とスタイル情報を取得すること
- 必要な画像を `src/img/{page}/` にダウンロードすること（SVGは `format=svg`、写真は `format=png`）
- ダウンロード後 `file` コマンドで実際の形式を確認すること（PNG偽装に注意）
- 結果を `docs/mcp-log/{page}/{section}.md` に保存すること

#### MCPログの保存（必須）

取得した値は **必ず** `docs/mcp-log/` に保存する。
ルール: [mcp-log/README.md](../mcp-log/README.md) を参照

### 2. セクションごとにコーディング

各セクションを順番に、**HTML → SCSS の順**でコーディングする。
**ページ全体を一度にコーディングしない。必ずセクション単位で進める。**

#### 各セクションの手順（1セクションごとに繰り返す）

1. **計画書を確認する**
   - `docs/mcp-log/{page}/{section}-plan.md` を開き、HTML構造とSCSS方針を確認
   - 計画書がない場合は、パターンライブラリから直接参照

2. **Figmaから該当セクションのデザインを取得（サブエージェント経由）**
   - Task ツールでサブエージェントを起動し、該当セクションの `get_design_context` を実行させる
   - サブエージェントが結果を `docs/mcp-log/` に保存後、メインはファイルを Read して作業する
   - 各セクションのノードを個別に選択して、テキスト内容やレイアウトを正確に確認

3. **画像の取得**
   - セクション内で使用される画像（サムネイル、アイコン、背景画像等）をFigma REST APIで取得
   - `src/img/{page}/` に保存
   - SVGアイコンは `format=svg`、写真系は `format=png` で取得
   - **複合背景の判断**: 3層以上のマスク/ブレンドを含む背景グループは、個別レイヤーではなくグループ単位で1枚画像として書き出す（詳細は [CODING_RULES.md](../CODING_RULES.md) §11 を参照）
   - ユーザーが `この背景はnode-id=xxxで1枚画像として取得して` と指示した場合はそれに従う

4. **HTMLの作成**
   - メインHTMLファイルに該当セクションのHTMLを追加
   - [CODING_RULES.md](../CODING_RULES.md) のHTML規約を遵守
   - **計画書のネスト図に従ってクラス名を付ける**

5. **SCSSの作成**
   - SCSSモジュールディレクトリに該当セクションのSCSSファイルを作成
   - [CODING_RULES.md](../CODING_RULES.md) のSCSS規約を遵守
   - **計画書のSCSS方針に従ってレイアウトを組む**

6. **セクション単位で目視確認**
   - Playwrightで該当セクションのスクリーンショットを取得（`--selector` で指定）
   - 明らかなレイアウト崩れや色の違いがないか確認
   - 大きな問題があればその場で修正
   - ※ pixelmatch比較は行わない（工程6で必要な箇所のみ実施）

7. **次のセクションへ進む**
   - 明らかな問題がなければ次のセクションへ進む

### ⚠️ 修正・調整時のバッチ実行ルール

複数の修正を行う場合、**全修正をまとめて実施してから、最後に1回だけビルド＋スクリーンショット確認する。**

```
❌ NG（修正ごとにビルド＋確認）:
  修正A → ビルド → スクショ確認 → 修正B → ビルド → スクショ確認 → ...
  （ビルド＋確認のオーバーヘッドが修正数に比例して増大）

✅ OK（バッチ実行）:
  修正A + 修正B + 修正C → ビルド → スクショ確認（1回のみ）
  （全修正をSCSS/HTMLに反映してから、まとめて検証）
```

**手順**:
1. 修正リストを確認し、全修正項目を把握する
2. 全修正のSCSS/HTML変更を一括で行う
3. `npx gulp build` を1回実行
4. Playwrightで該当セクションのスクリーンショットを1回取得
5. 全修正項目がまとめて反映されていることを確認

**例外**: レイアウトに大きな影響を与える修正（inner幅変更等）は、他の修正に影響するため先に単独で確認してもよい。

### ⚠️ SVG・アイコンの自作禁止（重要）

**SVGアイコンやイラストをインラインで自作してはならない。必ずFigmaからダウンロードすること。**

#### なぜ禁止か

- 自作SVGはFigmaオリジナルと形状が異なり、ピクセルパーフェクトを達成できない
- チェックボックス、電球、矢印など「簡単そうに見えるアイコン」ほど差異が目立つ

#### ダウンロード方法

1. **`get_design_context` のアセットURL を使う（推奨）**
   - `get_design_context` が返す `const imgXxx = "https://..."` のURLからダウンロード
   - `curl -L -o src/img/{page}/{filename}.svg "{URL}"`
   - ダウンロード後 `file` コマンドで実際の形式を確認（PNG偽装に注意）

2. **Figma REST API を使う（ノードIDが判明している場合）**
   ```bash
   # .env の FIGMA_ACCESS_TOKEN を使用
   # SVG形式でダウンロード
   curl -H "X-Figma-Token: $FIGMA_ACCESS_TOKEN" \
     "https://api.figma.com/v1/images/{fileKey}?ids={nodeId}&format=svg" \
     | jq -r '.images["{nodeId}"]' \
     | xargs curl -L -o src/img/{page}/{filename}.svg
   ```
   - 参考スクリプト: `docs/scripts/download-figma-assets.cjs`

3. **特定のアイコンのノードIDがわからない場合**
   - `get_metadata` でセクション内のノード一覧を取得
   - アイコン名（例: "weui:arrow-outlined"）を特定
   - そのノードIDで REST API を叩く

#### ダウンロードできない場合の対処

どうしてもダウンロードできない場合は、**コーディングを中断してユーザーに報告**する。
自作SVGで代替しない。

---

### 3. インスタンス（コンポーネント）から情報を取得する場合

`get_metadata` ではインスタンス内のテキスト内容は取得できない。
`get_design_context` を使用すること。

**注意:**
- `get_design_context` が返すReact + Tailwindのコードは参考情報として使用
- 実際のコーディングでは、プロジェクトの規約（[CODING_RULES.md](../CODING_RULES.md)）に従って実装

## デザイン再現性チェック

### 実装前の確認手順

1. **`get_screenshot` で視覚的に確認**
   - MCPからコード情報を取得する前に、必ずスクリーンショットを取得
   - 全体のレイアウトと細部のスタイルを目視で把握

2. **テキストスタイリングを個別にチェック**
   - 文中の **太字** 箇所を特定
   - 文中の **色違い** 箇所を特定
   - 同じ行内でも複数のスタイルが混在している可能性を考慮

### よくある見逃しパターン

| パターン | 原因 | 対策 |
|---------|------|------|
| テキストの一部だけ太字 | MCPのコード情報にスタイル詳細が含まれない | スクリーンショットで確認 |
| テキストの一部だけ色違い | 一般化してしまう | 個別に色を確認 |
| 複数スタイルの組み合わせ | `<span>`を1つしか付けない | 各スタイルに対応した`<span>`を入れ子にする |

## Playwrightによるスクリーンショット確認

コーディング後、Playwrightでスクリーンショットを取得し、Figmaデザインと比較する。

```bash
# 基本的な使用方法（SPビューポートでフルページキャプチャ）
node docs/scripts/screenshot.cjs

# 特定セクションのみキャプチャ
node docs/scripts/screenshot.cjs --selector ".l-footer" --viewport sp
node docs/scripts/screenshot.cjs --selector ".l-header" --viewport pc

# SP/PC両方をキャプチャ
node docs/scripts/screenshot.cjs --selector ".l-footer" --viewport both

# Figmaデザインと比較
node docs/scripts/screenshot.cjs --selector ".l-footer" --compare docs/screenshots/design.png
```

### オプション一覧

| オプション | 説明 | デフォルト値 |
|------------|------|-------------|
| `--url` | 対象URL | 自動検出 |
| `--selector` | キャプチャするCSSセレクタ | なし（全体） |
| `--viewport` | ビューポート（sp, pc, both） | sp |
| `--output` | 出力ディレクトリ | 自動設定 |
| `--name` | 出力ファイル名のプレフィックス | screenshot |
| `--compare` | 比較対象のFigmaデザイン画像パス | なし |
