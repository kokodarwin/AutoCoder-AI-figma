# Webデザイン実装パイプライン

**ミッション**: Figmaデザインを90%以上のピクセルパーフェクト精度でセマンティックHTML/SCSSに実装する

---

## 対象デザインの適正範囲

| 対象 | 適合度 | 備考 |
|------|--------|------|
| コーポレートサイト | OK | このワークフローの想定ターゲット |
| ランディングページ | OK | セクション単位の実装に適合 |
| ブログ・コラムページ | OK | テンプレート的な構造 |
| position:absolute多用サイト | NG | 原則禁止ルールと相性が悪い |
| 複雑なアニメーションサイト | NG | 別ワークフローを検討 |

## 前提条件

- Figma MCP が接続されていること
- Figma で対象のデザインが選択されていること
- プロジェクトのセットアップが完了していること（Vite/Gulp、SCSS環境など）
- 開発サーバーが起動中であること
- **`docs/agent.md`（エージェント知識ベース）を読み込み済みであること**（過去セッションの落とし穴・改善パターンを記録）

## コンテキスト隔離ルール（必須）

**Figma MCPの大量データによるコンテキスト圧縮を防止するため、以下を必ず守ること。**

### Figmaデータ取得はサブエージェントで実行する

`get_design_context` / `get_metadata` / `get_screenshot` は、**必ず Task ツール（サブエージェント）経由で実行する。**
メインエージェントがこれらを直接呼び出すことは禁止。

```
メインエージェント:
  1. Task ツールでサブエージェントを起動
  2. サブエージェントが Figma MCP を呼び出し、結果を docs/mcp-log/ に保存
  3. メインエージェントは保存されたファイルを Read で読んで作業する

サブエージェント（独立コンテキスト）:
  - get_design_context / get_metadata を実行
  - 必要な画像をダウンロード
  - 結果を docs/mcp-log/{page}/ に保存
  - 完了サマリーだけメインに返す
```

**理由**: Figma MCPのレスポンス（React+Tailwindコード、スクリーンショット、メタデータ）は1回あたり大量のトークンを消費する。
これがメインコンテキストに流入すると自動圧縮が発生し、パイプラインのルールや CODING_RULES.md の内容が失われる。
サブエージェントは独立したコンテキストを持つため、Figma生データはサブエージェント内に閉じ込められ、メインに漏れない。

### サブエージェントへの指示テンプレート

```
以下のFigmaノードからデザイン情報を取得し、docs/mcp-log/{page}/{section}.md に保存してください。

- get_design_context でデザイン構造とスタイル情報を取得
- get_metadata でレイヤーツリーを取得
- 必要な画像を src/img/{page}/ にダウンロード
- 結果は docs/mcp-log/ の保存ルール（docs/mcp-log/README.md）に従って記録

PC: {figma_url_pc}
SP: {figma_url_sp}
fileKey: {fileKey}
```

## レスポンシブ対応ルール

| 提供されたURL | PC実装 | SPレスポンシブ | SP pixel-perfect検証 |
|---|---|---|---|
| PCのみ | フル実装+検証 | 基本レスポンシブのみ（経験則） | スキップ |
| PC + SP | フル実装+検証 | Figma準拠で実装 | フル検証 |

- SP Figma URLが提供されていない場合、SPスクリーンショット撮影・pixelmatch比較は不要
- ただし `@include mq("md")` による基本レスポンシブ（フォントサイズ縮小、1カラム化等）は常に実装する

## 工程一覧（8フェーズ）

```
工程0 デザインシステム構築（フォント・カラー変数）
  ↓
工程1 画像エクスポート（コーディング時に自動取得 + ダウンロード後検証）
  ↓
工程2 Figmaレイヤー分析（レイヤー構造の事前分析）
  ↓
工程3 実装計画（HTML/SCSS方針の事前決定）
  ↓
工程4 コーディング（計画に基づくHTML→SCSS実装）
  ↓
工程5 セルフレビュー（規約・計画との照合 + クイック視覚チェック）
  ↓
  ✅ コーディング完了 → ユーザーに報告
  ↓
工程6 視覚的検証（★オンデマンド：ユーザーがmdファイルを渡して指示）
  ↓
工程7 品質チェック（エディタ内で完結）
```

### コアサイクル（自動実行）

```
Plan(3) → Implement(4) → Review(5) → Fix → 完了報告
```

工程0〜5はAIが自動進行する。工程5のクイック視覚チェックで明らかな差異があれば
自動で修正→再レビューを繰り返す。工程5完了時点でコーディング完了とする。

### オンデマンド検証（ユーザー指示で実行）

```
ユーザーが PIXELDIFF.md を渡す → pixelmatch検証 → RCA+Fix
```

工程6はユーザーがmdファイルを渡したときにのみ実行する。
毎回自動実行すると重いため、ユーザーが精密検証が必要と判断した場合にオンデマンドで実行する。

## 絶対ルール

### 0. Figma MCP接続の確認（最優先・違反時は即中断）

**Figma公式MCPと接続されていない場合、一切のコーディング作業を行わないこと。**

- コーディング開始前に必ずFigma MCPの接続状態を確認する
- 接続されていない場合は、**即座に作業を中断**し、ユーザーに「Figma MCPが接続されていません」と通知する
- 接続が確認できるまで、推測や過去の情報に基づいてコーディングを進めることは**厳禁**
- このルールは他のすべてのルールに優先する

```
IF figma_mcp_not_connected:
  STOP all coding immediately
  NOTIFY user: "Figma MCPが接続されていません。接続を確認してください。"
  WAIT for user confirmation
  DO NOT proceed under any circumstances
```

### 1. MCPログの保存（スキップ厳禁・精度厳守）

Figma MCPで取得したデザイン情報は、**コーディングに着手する前に必ず** `docs/mcp-log/{page}/design-context.md` に保存する。

```
手順:
1. get_design_context で情報を取得
2. docs/mcp-log/{page}/ に保存
3. 保存したファイルの内容を確認してから実装へ
```

#### MCPログ必須記録項目（1つでも欠けたらコーディング禁止）

以下の項目は `get_design_context` の出力から**正確な値を省略せず**に記録すること。
「白」「青系」など自然言語での要約は**厳禁**。必ずHEXコードや数値で記録する。

| カテゴリ | 必須記録項目 | NG例 → OK例 |
|----------|------------|-------------|
| グラデーション | 角度(deg)、各色(HEX)、各停止位置(%) | `青→濃い青` → `103.67deg, #3e9ccd 0.15%, #1e7aaa 98.4%` |
| 背景色 | 正確なHEXコード | `白` → `#eef6fa` |
| gap/padding/margin | px値（小数点含む） | `適度な間隔` → `gap: 28px` |
| テキストweight | 各テキストノードごとにweight値と適用範囲を明記 | `Bold混在` → `「〜強化したい」まで700、「など、」以降400` |
| 要素の座標 | top/left/right/bottom（親基準） | `右下に配置` → `left: 246px, top: 130px` |
| border-radius | px値 | `角丸` → `6px` |
| box-shadow | 完全な値 | `影あり` → `0 0 30px rgba(6,56,82,0.08)` |
| opacity | 小数値 | `薄い` → `0.1` |

### 2. データ不完全時の中断

`get_metadata` や `get_design_context` のレスポンスで、**明らかに要素が足りない場合**（例: デザイン上は6個あるのに4個しか返ってこない、テキストが途中で切れている等）は、**即座に実装を中断**する。

勝手に推測して実装を進めることは**厳禁**。必ずユーザーに報告し、完全な再取得を試みるか指示を仰ぐこと。

### 3. 画像取得のデフォルト

**コーディング時に必要な画像は自動的に取得する。**

- セクションのコーディング開始時に、そのセクションで使用される画像をFigma REST APIで取得する
- `get_design_context` が返すアセットURLや、ノード内のIMAGE fillの `imageRef` を使って画像をダウンロード
- ダウンロード先: `src/img/{page}/`（ページ単位で1フォルダ。例: `src/img/top/`, `src/img/concept/`）
- ダウンロード後は `file` コマンドで実際の形式を確認（PNG偽装に注意）

### 4. Figmaの値を正確に使う

- 推測や丸め込みをしない
- 小数点以下は四捨五入する
- 要素ごとに個別設定（一括指定は誤差の原因）

### 5. 計画書の作成（工程3）

コーディング前に必ず実装計画を作成し、mcp-logに保存する。
- パターンライブラリから適用パターンを選定
- HTML構造とSCSS方針を事前に決定
- 計画書なしにコーディングを開始しない

### 6. MCPログ参照ファースト（修正・調整時は必須）

**修正・調整セッションでは、Figma MCPに新規アクセスする前に `docs/mcp-log/` の既存ログを必ず確認する。**

初回コーディング時に取得済みのデザイン情報（HEXコード、px値、レイアウト等）はMCPログに正確に記録されている。
修正時にこれを無視してFigma MCPを再呼び出しすると、同じ情報の再取得で大量のトークンを浪費する。

```
手順:
1. docs/mcp-log/{page}/ 内のログファイルを確認
2. 必要な情報がログに存在する → ログの値を使用
3. ログに存在しない情報のみ → Figma MCPで取得してログに追記
```

### 7. 複雑なSVGのダウンロード禁止（トークン節約）

**マスク・複数VECTORレイヤーで構成された複雑なSVGイラストは、Figma MCPでダウンロード・合成を試みない。**

Figma MCP の `get_design_context` は複雑なSVGを個別のレイヤーに分解して返す。
これらを手動で合成しようとすると、1つのイラストで数万トークンを消費し、成功率も低い。

```
IF 複雑なSVG検出（マスク、3層以上のVECTOR、illust/character/person名）:
  1. ダウンロードを試みない
  2. SVG以外のコーディングを先に完了する
  3. コーディング完了報告時に、ユーザーに手動エクスポートを依頼:
     「以下のSVGはFigmaから手動エクスポートをお願いします:
      - {名前} (node-id: {ID}) → src/img/{page}/{filename}」
  4. ユーザーがエクスポートした後、コードに組み込む
```

詳細: [ASSETS.md - 複雑なSVGの取り扱い](./01-assets/ASSETS.md#複雑なsvgの取り扱いトークン節約)

### 8. agent.md への自動書き込み（知識の自動蓄積）

作業中に以下のいずれかに該当する発見があった場合、**ユーザーの指示を待たず**に `docs/agent.md` へ即座に追記する。

#### 書き込みトリガー

| トリガー | 例 |
|----------|-----|
| バグ修正で根本原因が非自明だった | 祖先のpadding-inlineがspace-betweenのgapを消していた |
| Figma→CSS変換で落とし穴があった | mix-blend-modeがREST APIエクスポートに反映されない |
| ワークフローの手順不足で問題を見落とした | クイック視覚チェックがなく画像重複に気づけなかった |
| ビルド・環境で予想外の動作があった | Gulpのwatch中にSCSS変更が反映されないケース |
| 同じ種類のミスを2回以上した | letter-spacingのem計算を間違えた等 |

#### 書き込みフォーマット

```
### 落とし穴: [簡潔なタイトル]

[何が起きたか（具体的な状況）]

**実例**: [実際のコード・値・セレクター名を含む再現可能な記述]

**対策**:
1. [防止手順1]
2. [防止手順2]
```

#### ルール

- **抽象的な教訓は書かない**。必ず具体的な実例（セレクター名、CSS値、計算式など）を含める
- 既存のエントリと重複する場合は、新規追加ではなく既存エントリに実例を追加する
- 書き込み後、ユーザーに「agent.mdに〇〇を追記しました」と1行で報告する

#### 自動分割ルール（肥大化防止）

agent.md の行数に応じて、以下を**自動実行**する。

| 行数 | アクション |
|------|-----------|
| ~500行 | 通常運用。そのまま追記を続ける |
| 500行超 | 追記時に「⚠️ agent.md が{N}行です。1000行超で自動分割します」と報告 |
| 1000行超 | **即座に自動分割を実行**（下記手順） |

##### 自動分割の手順（1000行超で発動）

```
1. agent.md の全セクションをトピック別に分類:
   - agent-mcp.md      … MCP設定・トークン管理・Figma API関連
   - agent-css.md       … CSS実装の落とし穴・レイアウト計算
   - agent-image.md     … 画像ダウンロード・エクスポート関連
   - agent-workflow.md  … ワークフロー構造・検証方法論
   - agent-env.md       … 開発環境・ビルド関連
   ※ トピックが上記に収まらない場合は適切な名前で新規ファイルを作成

2. agent.md を目次ファイル（インデックス）に書き換え:
   ---
   # Agent Knowledge Base - Index

   各トピックの詳細は個別ファイルを参照。

   | ファイル | 内容 | 行数 |
   |----------|------|------|
   | [agent-css.md](./agent-css.md) | CSS実装の落とし穴 | {N}行 |
   | [agent-image.md](./agent-image.md) | 画像関連 | {N}行 |
   | ... | ... | ... |

   ## 更新ルール
   [既存の更新ルール・トリガーをここに残す]
   ---

3. 分割後の読み込みルール:
   - セッション開始時は agent.md（インデックス）のみ読む
   - 各工程で関連トピックのファイルだけを読む
     例: 工程4（コーディング）→ agent-css.md, agent-image.md
     例: 工程1（画像エクスポート）→ agent-image.md, agent-mcp.md

4. 分割後の追記ルール:
   - 新しいエントリは該当トピックのファイルに追記する
   - agent.md（インデックス）の行数カラムを更新する

5. 完了後、ユーザーに報告:
   「agent.md が{N}行に達したため、{M}ファイルに自動分割しました」
```
